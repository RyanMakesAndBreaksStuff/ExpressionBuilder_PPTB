Initial Plan Review R1

Review target: docs/plans/initialplan.html. Review date: 2026-06-13. Requested skills: Power Automate Expression Builder and React Code Review.

Verdict: the plan is directionally valid, but it is not implementation-ready until the critical findings below are fixed.

The core direction is sound: a framework-free TypeScript expression engine, React web-first UI, and platform adapter boundary match the earlier architecture decision. The basic expression examples for Trigger Condition and Filter array advanced mode are valid. The main gaps are around boolean-root validation, type-aware expression generation, date handling, RAQB/Fluent UI dependency reality, and sharing the React app between web and PPTB without cross-app imports.

Critical Issues

CriticalShared React app has no package boundary

Plan references: apps/web/src/App.tsx at initialplan.html:531, apps/pptb/src/main.tsx at initialplan.html:604, PPTB wiring at initialplan.html:624, and "same React app" acceptance at initialplan.html:634.

The plan says web and PPTB boot paths render the same React app, but Wave 0 only creates apps/web, apps/pptb, packages/engine, and packages/platform. That leaves no clean place for shared UI. The likely implementation would either import source from apps/web into apps/pptb or duplicate screens, both of which violate the stated boundary.

Fix: add a shared UI package in Wave 0, for example packages/app or packages/builder-ui. Move AppShell, builder components, import/export, preview, and copy UI there. Keep apps/web and apps/pptb as thin bootstrap packages that pass different PlatformAdapter implementations.

CriticalPredicate roots are not constrained to boolean output

Plan references: AST root can be any ExpressionNode at initialplan.html:371-376, function nodes allow arbitrary calls at initialplan.html:391-395, literals include non-boolean values at initialplan.html:402-405, and required behavior only says root expressions get one leading @ at initialplan.html:492.

Trigger conditions and Filter array advanced mode predicates must evaluate to boolean. The plan exposes string/date-returning wrappers such as formatDateTime, utcNow, and addDays at initialplan.html:487, but does not require diagnostics when one of those functions, a field reference, or a literal becomes the root expression.

Fix: add expression return-type metadata and require formatExpression() to emit an error diagnostic for non-boolean roots in triggerCondition and filterArray modes. Add tests for invalid roots such as utcNow(), triggerBody()?['Title'], 'text', and 42.

CriticalExpression operators are not type-gated

Plan references: field types include number, boolean, dateTime, and choice at initialplan.html:361; empty checks are included generically at initialplan.html:485; sample fields include all those types at initialplan.html:571.

empty() is safe for null, strings, arrays, and objects, but it is not a general-purpose check for numbers and booleans. The plan also lists comparison operators for all fields without specifying date handling. Date comparisons should be explicit, usually through ticks() or an ISO-8601-only contract, otherwise lexicographic string comparison can be wrong.

Fix: make the operator registry field-type aware. Disallow or warn on empty() for numeric and boolean fields. Add date fixtures for @greater(ticks(triggerBody()?['DueDate']), ticks(utcNow())) and the equivalent item() form, or document and enforce a strict sortable timestamp format.

CriticalRAQB Fluent dependency conflicts with a Fluent UI v9 assumption

Plan references: tech stack lists Fluent UI and @react-awesome-query-builder/fluent at initialplan.html:271, the canvas implementation uses RAQB Fluent at initialplan.html:575, and release hardening pins RAQB at initialplan.html:759.

Current npm metadata still reports @react-awesome-query-builder/fluent as 6.7.0-alpha.0. Its peer dependencies are Fluent UI v8 packages: @fluentui/react, @fluentui/font-icons-mdl2, and @fluentui/react-icons. The current Fluent UI v9 package is @fluentui/react-components. If the implementation assumes Fluent UI v9 everywhere, the RAQB Fluent package does not cleanly match that design system.

Fix: explicitly choose one of two paths before implementation: use Fluent UI v8 for the MVP shell because RAQB Fluent expects it, or isolate RAQB v8 styling inside the builder surface while using v9 elsewhere with documented theme bridging. Keep the exact RAQB pin and verify install, render, and CSS behavior in Wave 0 or an explicit dependency spike.

CriticalRoot TypeScript build gate is underspecified

Plan references: Wave 0 creates tsconfig.base.json at initialplan.html:323, but no root tsconfig.json. The root script runs tsc -b at initialplan.html:348, and the Wave 0 gate expects package references to typecheck at initialplan.html:450.

tsc -b needs a root project file and project references, or it has no reliable build graph for package ordering and declarations. The plan also does not require composite, declaration output, package types, or package exports, all of which matter once workspaces import @pavb/engine, @pavb/platform, and the shared app package.

Fix: add root tsconfig.json, package-level composite configs, project references, declaration emit settings, and package exports/types requirements to Wave 0 before parallel work starts.

Warnings

WarningNull-safe string wrappers are not required

Plan reference: trigger formatter tests include @contains(toLower(triggerBody()?['Title']), 'urgent') at initialplan.html:503.

If Title is missing or null, safe navigation returns null and toLower() receives a non-string value. The plan should either emit a diagnostic for nullable wrapper input or generate a deliberate null-safe pattern such as @contains(toLower(coalesce(triggerBody()?['Title'], '')), 'urgent').

WarningCase sensitivity is not locked down in acceptance tests

Plan references: text operators are listed at initialplan.html:483, and the only case-insensitive example lowercases the field at initialplan.html:503.

Power Automate text comparisons and Filter array text filtering are case-sensitive unless the expression deliberately normalizes case. The plan should add exact-case fixtures and case-insensitive fixtures that normalize both the dynamic value and the user-entered value.

WarningSaved JSON stores two sources of truth

Plan reference: saved JSON includes both raqbTree and engineAst at initialplan.html:563-564.

Persisting both the RAQB tree and engine AST can drift. If they disagree, import/export round trips can silently change the generated expression or mask a converter bug.

Fix: persist the app-owned shape plus one canonical query representation, then derive the engine AST on load. If both must be stored for debugging, include a schema rule that validates both representations produce the same expression before import succeeds.

WarningESLint config target is stale for current latest ESLint

Plan reference: Agent D owns .eslintrc.cjs at initialplan.html:652 while the root script runs eslint . at initialplan.html:347.

Current npm metadata reports ESLint 10.5.0. Modern ESLint defaults to flat config, so a new project using latest ESLint should use eslint.config.js or explicitly pin an older ESLint version that still uses .eslintrc.cjs cleanly.

Fix: update Wave 0 and Agent D ownership to use eslint.config.js, or pin ESLint intentionally and document why.

WarningRAQB pinning happens too late

Plan references: Wave 0 runs npm install at initialplan.html:449, Agent B implements RAQB at initialplan.html:575, but the exact RAQB pin is not required until release hardening at initialplan.html:759.

The project should not discover RAQB alpha or peer dependency behavior after most implementation work is complete.

Fix: move exact RAQB and peer package pinning into Wave 0 or the first Agent B task, and make the dependency spike a merge gate before broad UI work.

WarningField references assume only top-level payload properties

Plan references: trigger fields render as triggerBody()?['FieldName'] at initialplan.html:493, and Filter array fields render as item()?['FieldName'] at initialplan.html:494.

Those references are valid for direct properties, but many trigger/action payloads need nested paths or trigger output metadata. Without field path metadata, the builder can only safely claim direct-property support.

Fix: add path or segments metadata to FieldDefinition, keep name as the stable field id, and document that MVP sample fields are direct properties unless metadata says otherwise.

WarningE2E persistence claim has no matching test step

Plan references: E2E acceptance says tests prove persistence at initialplan.html:690, but the E2E coverage list at initialplan.html:657-661 only covers mode switching, import/export, copy, and PPTB bridge calls.

Fix: add a web adapter settings persistence test and a PPTB settings persistence test, or remove "persistence" from the E2E acceptance criteria until settings are in scope.

WarningPPTB E2E target is not served

Plan references: Agent D creates tests/e2e/pptb-smoke.spec.ts at initialplan.html:647, but Playwright only starts npm run dev:web before tests at initialplan.html:673.

A PPTB smoke test needs a page to load. Starting only the web dev server does not prove the PPTB bundle, entrypoint, or mocked window.toolboxAPI boot path.

Fix: add a dev:pptb or preview:pptb target, or make the PPTB smoke test build apps/pptb and serve a static fixture with the toolbox API mock injected before app bootstrap.

WarningAccessibility is not part of acceptance

Plan references: UI requirements at initialplan.html:546-553 and E2E coverage at initialplan.html:657-661 do not require keyboard flow, focus management, labels, live regions for diagnostics, or segmented-control semantics.

A visual builder still needs keyboard-operable controls and accessible diagnostics. RAQB may supply some semantics, but the shell, mode selector, preview, import/export, and copy flow remain app responsibilities.

Fix: add Testing Library user-event tests for mode selection and copy/import/export keyboard flows, plus Playwright or axe checks for labels, focus order, and diagnostics announcements.

Suggestions

SuggestionStrengthen function registry acceptance before broad wrappers

Plan reference: function wrappers are included at initialplan.html:487, and function registry implementation is assigned at initialplan.html:509.

Add catalog metadata for parameter counts, parameter types, return type, allowed surfaces, and null behavior before exposing wrappers in the UI. This keeps formatDateTime, utcNow, and addDays from becoming free-form calls that are hard to validate.

SuggestionAdd a dependency spike to Wave 0

Plan reference: Wave 0 currently gates on package shells compiling at initialplan.html:293 and install/typecheck at initialplan.html:449-450.

Because RAQB Fluent is alpha and peers on Fluent UI v8, Wave 0 should include a minimal render spike that imports the RAQB Fluent builder in Vite, starts the web app, and captures one Playwright smoke screenshot before parallel work starts.

SuggestionAdd package-local scripts and an RAQB error boundary

The plan calls workspace-specific commands such as npm test --workspace @pavb/web, so each workspace package should explicitly define test, build, and where relevant dev/preview. Add an error boundary around the RAQB canvas and preview path because RAQB is the highest-risk UI dependency in the MVP.
