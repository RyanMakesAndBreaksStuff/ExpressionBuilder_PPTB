React Web-First Expression Builder Implementation Plan

Plan for implementing the recommended path from docs/reviews/3option.html: one pure TypeScript engine, one React web app, and one platform adapter boundary.

For agentic workers: REQUIRED SUB-SKILL: use superpowers:subagent-driven-development or equivalent task-by-task execution. This plan is intentionally split for multiple agents running in parallel after a short contract phase. Each agent has a disjoint ownership set and must not revert edits made by another worker.

Goal: build an MVP Power Automate visual expression builder that supports Trigger Condition and Filter Array modes, generates correct advanced-mode expressions, runs as a standalone React web app, and can later run inside Power Platform Toolbox through a thin adapter.

Architecture: the engine is a framework-free TypeScript package that receives a normalized AST and returns a Power Automate expression plus diagnostics. The React app uses @react-awesome-query-builder/fluent only as the input UI, then converts RAQB JSON into the engine AST. Web and PPTB capabilities are provided through a PlatformAdapter interface so app screens never call browser or PPTB APIs directly.

Tech Stack: npm workspaces, React 18, TypeScript, Vite, Vitest, Testing Library, Playwright, Fluent UI, @react-awesome-query-builder/fluent, GitHub Actions, GitHub Pages, and a future PPTB static package.

Elegant Boundary

Question asked: is there a more elegant way?

Answer: yes, keep the first implementation smaller than the drafts imply. The elegant version is not a hybrid codebase with UI forks. It is one core expression engine, one production React UI, and one platform adapter contract. Vue and Svelte stay out of the initial implementation because they do not reduce the hardest risk: formatter correctness.

Core Interfaces

Create these contracts before parallel work begins. Later agents may extend them only through additive changes.

// packages/engine/src/types.ts
export type ExpressionMode = "triggerCondition" | "filterArray";
export type FieldType = "string" | "number" | "boolean" | "dateTime" | "choice";
export type Conjunction = "and" | "or";

export interface FieldDefinition {
  name: string;
  label: string;
  type: FieldType;
  choices?: string[];
}

export type ExpressionNode =
  | GroupNode
  | RuleNode
  | FunctionCallNode
  | FieldReferenceNode
  | LiteralNode;

export interface GroupNode {
  kind: "group";
  conjunction: Conjunction;
  children: ExpressionNode[];
}

export interface RuleNode {
  kind: "rule";
  operator: string;
  left: ExpressionNode;
  right?: ExpressionNode;
}

export interface FunctionCallNode {
  kind: "function";
  name: string;
  args: ExpressionNode[];
}

export interface FieldReferenceNode {
  kind: "field";
  fieldName: string;
}

export interface LiteralNode {
  kind: "literal";
  value: string | number | boolean | null;
}

export interface FormatDiagnostic {
  code: string;
  message: string;
  severity: "error" | "warning";
  path: string;
}

export interface FormatterOptions {
  mode: ExpressionMode;
  fields: FieldDefinition[];
  maxDepth?: number;
}

export interface FormatResult {
  expression: string;
  diagnostics: FormatDiagnostic[];
}
// packages/platform/src/PlatformAdapter.ts
export type PlatformTheme = "light" | "dark" | "highContrast";
export type NotificationLevel = "success" | "info" | "warning" | "error";

export interface PlatformSettings {
  get(key: string): Promise<string | null>;
  set(key: string, value: string): Promise<void>;
  remove(key: string): Promise<void>;
}

export interface PlatformAdapter {
  copyToClipboard(text: string): Promise<void>;
  notify(message: string, level: NotificationLevel): Promise<void>;
  getTheme(): Promise<PlatformTheme>;
  onThemeChanged(handler: (theme: PlatformTheme) => void): () => void;
  settings: PlatformSettings;
  getDataverseFields?(): Promise<unknown[]>;
}

MVP Coverage

  • Equality: equals and not(equals(...)).
  • String checks: contains, startsWith, endsWith.
  • Comparisons: greater, less, greaterOrEquals, lessOrEquals.
  • Empty checks: empty and not(empty(...)).
  • Logic: nested and, or, and not.
  • Function wrappers: toLower, toUpper, trim, formatDateTime, utcNow, addDays.

Required Behaviors

  • Root expressions get exactly one leading @.
  • triggerCondition fields render as triggerBody()?['FieldName'].
  • filterArray fields render as item()?['FieldName'].
  • Strings escape single quotes as doubled single quotes, for example Bob''s Request.
  • Numbers, booleans, and null render without quotes.
  • Invalid ASTs return diagnostics instead of crashing, except documented programmer-error invariants.
  • Default max nesting depth is 10.

  3. Write literal tests for strings, escaped strings, numbers, booleans, dates as strings, and null.
  4. Write diagnostics tests for unsupported operator, missing right operand, unknown field, empty group, and max depth.
  5. Implement literal formatting in literals.ts.

Risk Register

Risk Impact Mitigation In This Plan
Expression generator produces subtly wrong Power Automate syntax. High Agent A owns exact formatter snapshots, diagnostics, and fixture-backed cookbook examples before UI polish.
RAQB Fluent package changes or alpha surface breaks integration. Medium to high Pin the package, isolate RAQB in apps/web/src/queryBuilder, and convert to engine AST immediately.
PPTB APIs drift or are unavailable in local development. Medium Keep all PPTB calls in pptbAdapter.ts and test against a mocked window.toolboxAPI.
Scope expands to broad function coverage before basic expressions are trusted. High MVP operator set is explicit; 140+ function coverage is deferred until snapshots are stable.
Parallel agents create overlapping edits. Medium Wave 0 creates contracts first; Wave 1 assigns disjoint ownership sets; integration happens after lane tests pass.
